# Diferencias con React y Next.js

RyunixJS **no es una copia de React** ni un metaframework encima de él. La API **recuerda** a React (componentes, JSX, hooks), pero el motor de renderizado es **propio**. Si conoces React o Next.js, gran parte de lo que ya sabes te sirve; lo que cambia son nombres, convenciones y lo que viene integrado de fábrica.

> [!WARNING]
> **Routing integrado** (rutas manuales con `RouterProvider`) ya no está soportado. Usa el **App Router** por carpetas bajo `app/`.

## Resumen para quien llega desde otro framework

| | React | Next.js | RyunixJS |
| :--- | :--- | :--- | :--- |
| **Qué es** | Librería de UI | React + full-stack | Motor de UI + full-stack **propio** |
| **Routing** | Externo (react-router, etc.) | App Router integrado | App Router integrado (`app/*.ryx`) |
| **Estado local** | `useState` | `useState` (React) | `useStore` |
| **SSR / SSG** | Manual o con librerías | Integrado | Integrado (`ryunix dev/build`) |
| **Server Actions** | No (en el core de React) | Sí | Sí |
| **Depende de React** | — | Sí | **No** |

**Ventaja clave de Ryunix:** obtienes una experiencia parecida a React + Next.js **sin arrastrar React** en el runtime. Menos capas, toolchain unificada bajo `@unsetsoft/*` y convenciones claras (`.ryx`, `ryunix.config.js`).

## Diferencias principales con React

### Eventos nativos del DOM

Ryunix **no usa eventos sintéticos**. Los handlers se registran con `addEventListener` del navegador, igual que en JavaScript puro:

```jsx
// React (eventos sintéticos)
<button onClick={(e) => console.log(e.nativeEvent)}>

// RyunixJS (eventos nativos)
<button onClick={(e) => console.log(e)}>
```

- Mejor rendimiento, sin capa extra de normalización
- Sin pooling de eventos
- El comportamiento sigue la especificación del DOM

### Nomenclatura de hooks

#### useState → useStore

```jsx
// React
import { useState } from "react";
const [count, setCount] = useState(0);
```

```jsx
// RyunixJS
import { useStore } from "@unsetsoft/ryunixjs";
const [count, setCount] = useStore(0);
```

El patrón es idéntico; el nombre deja claro que guardas estado del componente.

#### Hooks equivalentes a React

- `useEffect` — Efectos secundarios
- `useMemo` — Memorizar valores
- `useCallback` — Memorizar funciones
- `useRef` — Referencias
- `useReducer` — Patrón reducer
- `useTransition` — Transiciones
- `useDeferredValue` — Diferir actualizaciones

#### Hooks propios de Ryunix

Entre otros: `usePersistentStore` (sincroniza con `localStorage`), `useSwitch`, `useDebounce`, `useThrottle`, `useMetadata`.

### API de Context

Sintaxis distinta, mismo propósito:

```jsx
// React
const ThemeContext = createContext("light");
const theme = useContext(ThemeContext);
```

```jsx
// RyunixJS
const { Provider, useContext } = createContext("theme", "light");
const theme = useContext();
```

1. `createContext` requiere un ID único
2. Devuelve `{ Provider, useContext }`
3. `useContext()` se llama sin argumentos

## App Router frente a react-router

En React puro sueles añadir **react-router** y definir rutas en código. En Ryunix (y en Next.js) las rutas salen de **carpetas y archivos**:

```
app/
├── layout.ryx
├── index.ryx          → /
└── about/
    └── index.ryx      → /about
```

Layouts, loading, errores y metadatos viven junto a cada segmento. Detalle en [Navegación](/es/docs/introduction/navigation).

## Lo que Ryunix trae integrado (y React no)

Estas capacidades forman parte del ecosistema oficial; con React solo suelen llegar vía Next.js u otras librerías:

| Capacidad | En Ryunix |
| :--- | :--- |
| **SSR y SSG** | `ryunix dev`, `ryunix build`; ver [Renderizado](/es/docs/core/rendering) |
| **Server Components** | Inferidos en compile time desde `app/` |
| **Server Actions** | Funciones `"use server"`; ver [Server Actions](/es/docs/core/server-actions) |
| **Rutas API** | `app/api/**/router.js` |
| **MDX nativo** | Páginas `.mdx` en el App Router |
| **Metadatos SEO** | `Metatags`, `generateMetadata`, `useMetadata` |

### Metadatos en páginas

```jsx
export const Metatags = {
  title: "Mi página",
  description: "Descripción para buscadores",
};

export default function Page() {
  return <main>Contenido</main>;
}
```

## No disponible (respecto a React clásico)

- ❌ **Componentes de clase** — Usa componentes funcionales
- ❌ **PropTypes**
- ❌ **Modo concurrente completo** de React 18 (Suspense básico sí)
- ❌ **Ecosistema npm que exige `react` como peer** — sin adaptación

## Consejos rápidos de migración desde React

1. Cambia los imports de `react` a `@unsetsoft/ryunixjs` y `useState` → `useStore`.
2. Elimina `ReactDOM.createRoot`; en apps con App Router el entry lo genera el preset.
3. Sustituye react-router por carpetas en `app/`.
4. Actualiza Context al patrón `createContext(id, default)`.
5. Convierte componentes de clase a funcionales.

Consulta [Acerca de Ryunix](/es/docs/introduction/about) para el panorama general y [Configuración](/es/docs/api/configuration) para el setup de build.
