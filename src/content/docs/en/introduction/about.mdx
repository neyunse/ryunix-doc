# Introduction

**RyunixJS** is a JavaScript framework for building **websites and interactive applications** ŌĆö from a portfolio or documentation site to an admin dashboard or an app with login, forms, and server data.

If you already know some web basics (HTML, CSS, JavaScript), Ryunix gives you a modern way to build the UI **without wiring everything yourself**: routing, state, server rendering, and dev tooling are built in.

## What is it, in plain terms?

Ryunix is a **UI runtime** with a programming model **similar to React** (components, JSX, hooks), but **it does not use React internally**. It ships its own engine ŌĆö reconciler, hooks, and render pipeline ŌĆö designed to stay **lightweight, fast, and maintainable**.

In practice you write `.ryx` files with JSX, organize pages in folders under `app/`, and use commands like `ryunix dev`, `ryunix build`, and `ryunix start` to develop and ship.

## How does it work?

The usual flow is straightforward:

1. **Write components** ŌĆö functions that return UI (buttons, lists, forms).
2. **Define routes with folders** ŌĆö each folder under `app/` becomes a URL (`app/about/index.ryx` ŌåÆ `/about`).
3. **The CLI compiles your app** ŌĆö Webpack produces client and server bundles.
4. **The runtime renders the UI** ŌĆö in the browser, on the server (SSR), or as static HTML (SSG), depending on what you need.

You do not install React, react-router, or Next.js: the framework bundles engine, routing, and toolchain in `@unsetsoft/*` packages.

## Relationship with React and Next.js

Ryunix **is not React or Next.js**, but if you know them the learning curve is gentle.

### Coming from React

| Concept | React | RyunixJS |
| :--- | :--- | :--- |
| Components & JSX | Ō£ģ | Ō£ģ Same idea |
| Local state | `useState` | `useStore` (same pattern) |
| Effects, memo, refs | `useEffect`, `useMemo`, ŌĆ” | Equivalent hooks |
| Events | Synthetic system | Native DOM events |
| Routing | External library (e.g. react-router) | **Built-in App Router** (file-based) |

Ryunix **feels like** ReactŌĆÖs API, but the runtime is **its own**. You cannot drop in libraries that require `react` as a peer dependency without adapting them.

### Coming from Next.js

| Concept | Next.js | RyunixJS |
| :--- | :--- | :--- |
| File-based routing | `app/page.tsx`, `layout.tsx` | `app/index.ryx`, `layout.ryx` |
| Hybrid rendering | CSR, SSR, SSG | CSR, SSR, SSG |
| Full-stack | API routes, Server Actions | `app/api/.../router.js`, Server Actions |
| Commands | `next dev`, `build`, `start` | `ryunix dev`, `build`, `start` |
| UI engine | React | **Own engine** (`@unsetsoft/ryunixjs`) |

**Quick analogy:** React is the UI engine; Next.js is React + full-stack. In Ryunix, `@unsetsoft/ryunixjs` is the engine and `@unsetsoft/ryunix-presets` + Create Ryunix App provide the full-stack layer.

## What Ryunix offers on its own

### Standalone architecture

The core **does not embed React or Preact**. That means a smaller runtime, no dependency on third-party release cycles, and direct control over how the framework evolves.

### Familiar API, own conventions

Hooks like `useEffect` or `useMemo` feel like React; others are Ryunix-specific (`useStore`, `usePersistentStore`, `useSwitch`, `useDebounce`). Files use the `.ryx` extension and config lives in `ryunix.config.js`.

### Integrated full-stack

App Router, SSR, SSG, Server Components, Server Actions, API routes, and native MDX are part of the official stack ŌĆö not scattered optional plugins.

### Ready-to-use toolchain

Create Ryunix App scaffolds a project in seconds. The VS Code extension highlights `.ryx`, DevTools inspects the component tree, and the CLI unifies dev, build, and production.

### Performance and simplicity

Fiber-based reconciler, native DOM events, and a core with no external runtime dependencies. The philosophy favors **clarity and maintainability** over unnecessary abstraction.

## Good fit for

- Static sites, landings, and blogs (SSG)
- MDX documentation (like this site)
- SPAs and interactive dashboards
- Full-stack apps with APIs and server logic

## Philosophy

Modern frontend development often stacks complexity: many layers, coupled tooling, and frequent API churn. Ryunix aims for a more direct path:

> **Build modern web applications with full control and minimal complexity.**

With Ryunix you get explicit application logic, a predictable dev experience, and a foundation meant for long-lived projects.
