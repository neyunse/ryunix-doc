# Differences with React and Next.js

RyunixJS **is not a clone of React** or a metaframework on top of it. The API **feels like** React (components, JSX, hooks), but the rendering engine is **its own**. If you know React or Next.js, much of what you already know transfers; what changes are names, conventions, and what ships built in.

> [!WARNING]
> **Integrated Routing** (manual routes with `RouterProvider`) is no longer supported. Use the **App Router** with folders under `app/` instead.

## At a glance if you come from another framework

| | React | Next.js | RyunixJS |
| :--- | :--- | :--- | :--- |
| **What it is** | UI library | React + full-stack | Own UI engine + full-stack |
| **Routing** | External (react-router, etc.) | Built-in App Router | Built-in App Router (`app/*.ryx`) |
| **Local state** | `useState` | `useState` (React) | `useStore` |
| **SSR / SSG** | Manual or extra libs | Built in | Built in (`ryunix dev/build`) |
| **Server Actions** | Not in React core | Yes | Yes |
| **Depends on React** | — | Yes | **No** |

**Ryunix advantage:** you get a React + Next.js–like experience **without shipping React** in the runtime. Fewer layers, a unified toolchain under `@unsetsoft/*`, and clear conventions (`.ryx`, `ryunix.config.js`).

## Main differences from React

### Native DOM events

Ryunix **does not use synthetic events**. Handlers use the browser’s `addEventListener`, like plain JavaScript:

```jsx
// React (synthetic events)
<button onClick={(e) => console.log(e.nativeEvent)}>

// RyunixJS (native events)
<button onClick={(e) => console.log(e)}>
```

- Better performance, no extra normalization layer
- No event pooling
- Behavior follows the DOM specification

### Hooks nomenclature

#### useState → useStore

```jsx
// React
import { useState } from "react";
const [count, setCount] = useState(0);
```

```jsx
// RyunixJS
import { useStore } from "@unsetsoft/ryunixjs";
const [count, setCount] = useStore(0);
```

Same pattern; the name makes it explicit that you store component state.

#### Hooks equivalent to React

- `useEffect` — Side effects
- `useMemo` — Memoize values
- `useCallback` — Memoize functions
- `useRef` — Refs
- `useReducer` — Reducer pattern
- `useTransition` — Transitions
- `useDeferredValue` — Defer updates

#### Ryunix-specific hooks

Including `usePersistentStore` (syncs with `localStorage`), `useSwitch`, `useDebounce`, `useThrottle`, `useMetadata`.

### Context API

Different syntax, same purpose:

```jsx
// React
const ThemeContext = createContext("light");
const theme = useContext(ThemeContext);
```

```jsx
// RyunixJS
const { Provider, useContext } = createContext("theme", "light");
const theme = useContext();
```

1. `createContext` requires a unique ID
2. Returns `{ Provider, useContext }`
3. `useContext()` is called with no arguments

## App Router vs react-router

With plain React you usually add **react-router** and define routes in code. In Ryunix (and Next.js) routes come from **folders and files**:

```
app/
├── layout.ryx
├── index.ryx          → /
└── about/
    └── index.ryx      → /about
```

Layouts, loading, errors, and metadata live next to each segment. See [Navigation](/en/docs/introduction/navigation) for details.

## What Ryunix includes (that React alone does not)

These are part of the official stack; with React they typically come via Next.js or other libraries:

| Capability | In Ryunix |
| :--- | :--- |
| **SSR & SSG** | `ryunix dev`, `ryunix build`; see [Rendering](/en/docs/core/rendering) |
| **Server Components** | Inferred at compile time from `app/` |
| **Server Actions** | `"use server"` functions; see [Server Actions](/en/docs/core/server-actions) |
| **API routes** | `app/api/**/router.js` |
| **Native MDX** | `.mdx` pages in the App Router |
| **SEO metadata** | `Metatags`, `generateMetadata`, `useMetadata` |

### Page metadata

```jsx
export const Metatags = {
  title: "My page",
  description: "Description for search engines",
};

export default function Page() {
  return <main>Content</main>;
}
```

## Not available (vs classic React)

- ❌ **Class components** — Use functional components
- ❌ **PropTypes**
- ❌ **Full React 18 concurrent mode** (basic Suspense yes)
- ❌ **npm ecosystem packages that require `react` as a peer** — without adaptation

## Comparison table

| Feature | React | Next.js | RyunixJS |
| :--- | :--- | :--- | :--- |
| **State hooks** | `useState` | `useState` | `useStore` |
| **Events** | Synthetic | Synthetic (React) | Native |
| **Routing** | External | App Router | App Router |
| **SSR / SSG** | Manual | Built in | Built in |
| **Server Actions** | — | Yes | Yes |
| **MDX** | Plugin | Built in | Built in |
| **Engine** | React | React | **Own** |
| **Class components** | ✅ | ✅ | ❌ |

## Quick migration tips from React

1. Change imports from `react` to `@unsetsoft/ryunixjs` and `useState` → `useStore`.
2. Remove `ReactDOM.createRoot`; with the App Router the preset generates the entry.
3. Replace react-router with folders under `app/`.
4. Update Context to `createContext(id, default)`.
5. Convert class components to functional ones.

See [About Ryunix](/en/docs/introduction/about) for the big picture and [Configuration](/en/docs/api/configuration) for build setup.
